## Stigmergy 架构分析报告

### 一、架构概述

**Stigmergy** 是一个基于**痕迹刺激**（Stigmergy）概念的多 Agent CLI 协作系统，核心思想是通过共享的痕迹（文件、记忆、技能）实现 AI 助手之间的间接协作和自我进化。

### 三、架构优势 ✅

1. **技能驱动设计** - 50+ 预定义技能覆盖完整开发工作流
2. **双 Agent 循环机制** - 主 Agent + Subagent + Code Review 的自检循环
3. **自主进化能力** - 支持自我反思、知识提取、技能创建
4. **跨 CLI 协作** - 支持 Claude/Gemini/Qwen 等多 AI 工具协作
5. **痕迹持久化** - 通过文件实现状态持久化和跨会话记忆

### 四、识别的问题与改进建议

#### 🔴 高优先级问题

| 问题 | 影响 | 建议修复 |
|------|------|----------|
| **1. 技能路径混乱** | `skill_discovery_paths` 配置了 5 个不同路径，导致技能发现不可靠 | 统一为单一权威路径 `~/.stigmergy/skills/` |
| **2. Superpowers 路由问题** | `qwen_safe_executor.js` 阻止了 `stigmergy ... superpowers:` 命令，但安全技能又要求通过 stigmergy 路由 | 移除矛盾的安全阻止，改为白名单验证 |
| **3. Hook 配置不一致** | `hooks.json` 缺少 `forbidden_patterns` 和 `process_safety` 配置（与 `qwen_config_patcher.js` 生成内容不匹配） | 同步实际配置与生成器代码 |
| **4. 会话注入路径错误** | `session-start.ts` 使用 `SKILLS_ROOT = ~/skills` 而非 `~/.stigmergy/skills` | 修复路径常量 |
| **5. 自动记忆扩展无退出集成** | `auto-memory/index.js` 注册了 exit handler 但实际可能无法在 CLI 退出时触发 | 使用进程信号捕获 + 定期持久化 |

#### 🟡 中优先级问题

| 问题 | 影响 | 建议修复 |
|------|------|----------|
| **6. 技能版本控制缺失** | 技能文件没有版本追踪，无法回滚或追踪进化历史 | 添加 `version` 字段 + 变更日志 |
| **7. 双 Agent 循环未完全实现** | `soul-evolution` 描述的流程与实际 `two-agent-loop` 实现可能不匹配 | 审核并同步实现 |
| **8. 反思结果未利用** | `soul-reflection` 保存反思到 `~/.stigmergy/reflection/` 但无后续消费机制 | 添加反思驱动的自动改进触发器 |
| **9. 跨 CLI 通信单向** | 仅支持 `stigmergy distcli` 发出请求，缺少响应聚合机制 | 实现响应回调/轮询机制 |
| **10. 配置分散** | 配置分散在 `config.json`、`settings.json`、`hooks.json` 中 | 创建统一的 `stigmergy.config.json` |

#### 🟢 低优先级改进

| 改进 | 价值 |

| **12. 技能热重载** | 修改技能后无需重启 CLI |
| **13. 进化仪表板** | 可视化展示进化进度、技能增长、反思趋势 |
