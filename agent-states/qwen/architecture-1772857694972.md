## Stigmergy 架构分析报告

### 一、当前架构概览

**Stigmergy** 是一个基于**痕迹协调**（stigmergy）概念的多智能体 CLI 系统，核心思想是通过环境痕迹（文件、日志、技能）实现智能体间的间接协作。

#### 核心组件

| 组件 | 位置 | 功能 |
|------|------|------|
| **技能系统** | `~/.qwen/skills/` | 40+ 预定义技能，涵盖规划、测试、调试等 |
| **钩子系统** | `~/.qwen/hooks/` | Session 启动、安全执行、跨 CLI 通信 |
| **跨 CLI 通信** | `config.json` | 支持 8 种 CLI 工具互操作 |
| **Superpowers** | `extensions/superpowers-qwen/` | 高级功能封装 |
| **Agents** | `agents/agents/` | 专业领域智能体（ANT、场域分析等） |
| **Plugins** | `plugins/` | DNASPEC 认知模板插件 |

---

### 二、架构优势

1. **痕迹协调机制** - 通过文件系统实现智能体间异步协作
2. **技能发现系统** - 多路径技能搜索与自动刷新
3. **安全执行层** - 命令过滤、模式匹配、超时控制
4. **跨平台互操作** - 支持多种 CLI 工具协同工作
5. **自我进化能力** - `soul-auto-evolve` 和 `soul-reflection` 技能

---

### 三、架构问题与改进建议

#### 🔴 高优先级问题

| 问题 | 影响 | 建议 |
|------|------|------|
| **1. 技能加载机制不统一** | Qwen CLI 使用 `Skill` 工具，iFlow 使用文件读取，CodeBuddy 另有方式 | 建立统一的技能加载协议层 |
| **2. 跨 CLI 通信依赖 shell 命令** | `stigmergy distcli` 通过 shell 调用，存在注入风险 | 实现基于 MCP 的标准通信协议 |
| **3. 安全规则硬编码** | `qwen_safe_executor.js` 中安全规则固定，难以动态更新 | 建立可配置的安全策略引擎 |
| **4. 技能版本管理缺失** | 技能文件无版本控制，更新可能覆盖用户修改 | 引入技能版本追踪和冲突解决机制 |
| **5. 会话历史分散存储** | 历史数据存储在多个位置，难以聚合分析 | 统一历史存储到 `~/.stigmergy/history/` |

```
│                      Core Services                       │

│  └──────────┴──────────┴──────────┴──────────────────┘  │

│                      Storage Layer                       │
│  (~/.stigmergy/{skills,history,reflection,config}/)     │
└─────────────────────────────────────────────────────────┘
```

#### 方案 B：MCP 协议集成

将跨 CLI 通信从 shell 命令升级为 **Model Context Protocol**：

```typescript
// 当前方式（不安全）
execSync(`stigmergy distcli "${prompt}"`)

// 建议方式（MCP）
const mcpClient = new MCPClient('stigmergy://localhost');
await mcpClient.invoke('cross_cli.execute', { 
  target: 'claude', 
  prompt 
});
```

### 六、总结

Stigmergy 架构的核心理念（痕迹协调、自主进化）非常先进，但当前实现存在**安全、可维护性、扩展性**三方面的改进空间。建议优先解决安全执行和统一技能加载机制，然后逐步向分层架构和 MCP 协议迁移。
