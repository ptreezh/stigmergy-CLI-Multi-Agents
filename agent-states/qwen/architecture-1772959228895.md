## Stigmergy 架构分析

### 当前架构概览

**核心设计理念**：基于** stigmergy（痕迹协调）** 模式的多 Agent 协作系统，通过共享存储和痕迹信号实现 Agent 间的间接协调。

#### 核心组件

| 组件 | 位置 | 功能 |
|------|------|------|
| **Session Hook** | `hooks/session-start.ts` | 会话启动时注入 Superpowers 上下文、Soul 身份、进化状态 |
| **Superpowers 集成** | `skills/using-superpowers/SKILL.md` | 元技能，定义技能发现和调用机制 |
| **双 Agent 循环** | `skills/two-agent-loop/SKILL.md` | 基础执行机制：派发 Subagent → Code Review → 迭代修复 |
| **进化系统** | `skills/soul-*/SKILL.md` | 自主进化、协同进化、竞争进化 |
| **安全执行器** | `hooks/qwen_safe_executor.js` | 阻止未定义的 superpowers 调用 |
| **技能系统** | `skills/*/SKILL.md` | 模块化技能包，支持渐进式披露 |

---

### 架构优势 ✅

1. **双 Agent 循环设计** - 每个子任务都有质量审查，确保输出质量
2. **技能模块化** - SKILL.md + 资源文件的清晰结构
3. **渐进式披露** - 元数据 → 技能体 → 资源文件的三级加载
4. **安全机制** - 白名单过滤 superpowers 调用
5. **进化理念** - 反思 → 知识提取 → 技能创建的闭环

---

### 架构问题 🔴

#### 1. **循环依赖风险**

```
two-agent-loop (基础)
    ↓
soul-evolution (用双循环实现)
    ↓
soul-co-evolve (用双循环实现)
    ↓
soul-compete (用双循环实现)
```

所有进化技能都依赖 `two-agent-loop`，但 `two-agent-loop` 本身没有独立的验证机制。如果基础循环有问题，所有进化都会放大错误。

#### 2. **状态管理分散**

状态分散在多个位置，没有统一的状态机：

```
~/.stigmergy/
├── superpowers-version.json    # 版本状态
├── soul-state/
│   ├── evolutions/             # 进化记录
│   ├── evolved-skills/         # 创建的技能
│   ├── evolution-log.jsonl     # 进化日志
│   ├── co-evolution/           # 协同进化状态
│   └── competition/            # 竞争进化状态
├── reflection/{cli}/           # 反思记录
└── history/{cli}/              # 历史记录
```

问题：
- 没有原子性保证
- 状态同步困难
- 恢复机制缺失

#### 3. **安全模型过于简单**

`qwen_safe_executor.js` 使用正则表达式白名单：

```javascript
this.blockedCommands = [
  /superpowers:.*[^a-zA-Z0-9_-]/i,
  /stigmergy.*superpowers:.*/i,
  // ...
];

this.safeSuperpowers = [
  'using-git-worktrees',
  'writing-plans',
  // ... 重复列表
];
```

问题：
- 正则表达式可绕过
- 白名单硬编码，无法动态更新
- 没有权限分级

#### 4. **技能触发机制不清晰**

`using-superpowers` 说"1% 规则"，但没有自动化触发器：

> "IF A SKILL APPLIES TO YOUR TASK, YOU DO NOT HAVE A CHOICE. YOU MUST USE IT."

实际依赖 LLM 自觉加载技能，缺乏：
- 技能匹配算法
- 强制加载机制
- 技能使用审计

#### 5. **进化闭环未验证**

```
反思 → 知识提取 → 技能创建 → 验证测试
```

但：
- 没有定义"反思质量"标准
- 知识提取依赖 LLM 主观判断
- 验证测试可能流于形式

#### 6. **跨 CLI 协调是空壳**

`CROSS_CLI_GUIDE.md` 只定义了命令格式：

```bash
stigmergy distcli "Prompt"
```

但没有：
- 共享存储协议
- 知识同步机制
- 冲突解决策略

#### 1. **引入状态机管理**

```
┌─────────────────────────────────────────────────────────┐
│              Stigmergy State Machine                    │
├─────────────────────────────────────────────────────────┤
│  IDLE → PLANNING → EXECUTING → REVIEWING → EVOLVING    │
│    ↑                                              │    │
│    └──────────────────←───────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

每个状态：
- 定义进入条件
- 定义退出条件
- 定义允许的操作
- 原子性状态转换日志
```
