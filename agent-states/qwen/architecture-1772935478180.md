## Stigmergy 架构分析

### 核心架构概述

Stigmergy 是一个**基于双 Agent 循环的自我进化系统**，核心思想是通过 stigmergy（痕迹协调）机制实现多 CLI、多 Agent 之间的协作与进化。


## 发现的问题与改进建议

### 1. **存储路径不统一**

**问题**: 文档中存储路径分散且不一致：
- `~/.stigmergy/soul-state/` (soul 技能文档)
- `~/.stigmergy/skills/{cli}/` (soul-auto-evolve)
- `~/.stigmergy/reflection/{cli}/` (soul-auto-evolve)

**建议**: 统一存储结构：
```
~/.stigmergy/
├── state/
│   ├── evolutions/        # 进化记录
│   ├── reflections/       # 反思记录
│   ├── competitions/      # 竞争结果
│   └── co-evolution/      # 协同进化
│       ├── shared/
│       ├── received/
│       └── my-shared/
├── skills/                # 创建的技能
├── knowledge/             # 知识库
└── logs/                  # 执行日志
```

### 2. **缺少状态管理机制**

**问题**: 双 Agent 循环没有持久化状态，中断后无法恢复。

**建议**: 添加状态检查点：
```javascript
// ~/.stigmergy/state/loop-state.json
{
  "loopId": "uuid",
  "currentTask": "task-2",
  "completedTasks": ["task-1"],
  "pendingTasks": ["task-3", "task-4"],
  "reviewStatus": {
    "task-2": { "spec": "pending", "quality": "not-started" }
  },
  "lastCheckpoint": "2026-03-08T10:30:00Z"
}
```

### 3. **协同进化缺少实际通信协议**

**问题**: `soul-co-evolve` 提到多 CLI 协作，但没有定义：
- CLI 间如何发现彼此
- 知识如何同步（推送/拉取）
- 冲突如何解决

**建议**: 设计简单的通信协议：
```
┌─────────────────────────────────────────────────────────┐
│              协同进化通信协议                            │
├─────────────────────────────────────────────────────────┤
│  1. 注册：CLI 启动时向本地服务注册                        │
│  2. 广播：知识发布时广播给所有在线 CLI                    │
│  3. 订阅：CLI 可订阅特定类型的知识更新                    │
│  4. 同步：定时拉取离线期间的更新                          │
└─────────────────────────────────────────────────────────┘
```

### 4. **Code Review 质量依赖 LLM 主观判断**

**问题**: 当前 Code Review 完全依赖 LLM 判断，缺少：
- 客观检查清单
- 自动化测试验证
- 代码度量标准

**建议**: 添加检查清单和自动化验证：
```markdown
# Code Review 检查清单

## Spec 合规性
- [ ] 所有需求功能已实现
- [ ] 无额外未请求功能
- [ ] 接口签名匹配 spec

## 代码质量
- [ ] 测试覆盖率 > 80%
- [ ] 无 console.log 遗留
- [ ] 无硬编码魔法数字
- [ ] 错误处理完整

## 自动化验证
- [ ] 所有测试通过
- [ ] lint 检查通过
- [ ] 类型检查通过
```

### 5. **缺少进化效果评估**

**问题**: 无法衡量进化是否真正提升了能力。

**建议**: 添加评估指标：
```javascript
// 进化效果评估指标
const metrics = {
  // 效率指标
  taskCompletionTime: "平均任务完成时间",
  subagentDispatchCount: "单位任务 subagent 派发次数",
  reviewIterationCount: "平均 review 迭代次数",
  
  // 质量指标
  firstPassSuccessRate: "一次性通过率",
  bugRate: "单位代码 bug 数量",
  specComplianceRate: "spec 合规率",
  
  // 学习指标
  newSkillsCreated: "新增技能数",
  knowledgeReuseRate: "知识复用率",
  crossCliAdoptionRate: "跨 CLI 采用率"
};
```

### 6. **two-agent-loop 缺少并发控制**

**问题**: 文档说"不并行派发"，但 `soul-compete` 需要并行执行多个方案。

**建议**: 明确并发策略：
```
┌─────────────────────────────────────────────────────────┐
│              并发控制策略                                │
├─────────────────────────────────────────────────────────┤
│  顺序执行 (默认):                                        │
│    - 任务间有依赖关系                                    │
│    - 共享状态修改                                        │
│    - 资源竞争场景                                        │
│                                                         │
│  并行执行 (需显式声明):                                  │
│    - 方案对比 (soul-compete)                            │
│    - 独立数据收集                                        │
│    - 使用 git worktree 隔离                             │
└─────────────────────────────────────────────────────────┘
```

### 7. **技能触发机制不清晰**

**问题**: 各技能触发方式描述模糊：
- "关键词触发" - 谁来监听关键词？
- "Scheduler 定时" - 调度器在哪里？
- "手动触发" - 具体命令是什么？

**建议**: 明确触发机制：
```javascript
// hooks/skills/ 触发器配置
{
  "soul-reflection": {
    "triggers": ["反思", "reflect", "分析"],
    "scheduler": "0 * * * *",  // 每小时
    "command": "stigmergy skill soul-reflection"
  },
  "soul-evolution": {
    "triggers": ["进化", "evolve"],
    "scheduler": "0 0 * * 0",  // 每周日
    "command": "stigmergy skill soul-evolution"
  }
}
```

### 8. **缺少错误恢复机制**

**问题**: 如果 subagent 执行失败、LLM 超时、存储写入失败，没有恢复策略。

**建议**: 添加错误处理：
```
错误恢复策略:
├── Subagent 失败 → 重试 (最多 3 次) → 降级 (简化任务) → 人工介入
├── LLM 超时 → 切换模型 → 使用缓存 → 跳过非关键步骤
└── 存储失败 → 使用临时存储 → 队列延迟写入 → 告警
```
