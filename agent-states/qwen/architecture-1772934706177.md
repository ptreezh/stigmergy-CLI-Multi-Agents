## Stigmergy 架构分析报告

### 📋 架构概览

Stigmergy 是一个基于**痕迹刺激**（Stigmergy）理论的多 Agent CLI 协作系统，版本 1.2.1。核心理念是通过共享存储和异步信号实现多 CLI 间的协作进化。

---

### ✅ 架构优势

| 优势 | 说明 |
|------|------|
| **双 Agent 循环机制** | 主 Agent + Subagent 的串行执行模式，确保质量和可控性 |
| **技能系统** | 模块化的技能设计（40+ 技能），支持热插拔和自主进化 |
| **多 CLI 支持** | 支持 Claude、Qwen、Gemini 等 8 种 CLI 工具 |
| **自主进化** | 通过 soul-evolution 实现自我学习和技能创建 |
| **协同进化** | 多 CLI 间共享知识库，实现协作学习 |
| **Hooks 机制** | session-start、auto-memory 等钩子实现自动注入 |
| ** stigmergy 存储** | 统一的 `~/.stigmergy/` 目录存储历史、反思、技能 |

### ⚠️ 架构问题与改进建议

#### 1. **核心机制问题**

| 问题 | 严重性 | 改进建议 |
|------|--------|----------|
| **双 Agent 循环不支持并行** | 中 | 引入 `dispatching-parallel-agents` 到核心循环，允许独立子任务并行执行 |
| **Subagent 返回后缺乏状态管理** | 高 | 添加状态机管理子任务状态（pending/running/completed/failed） |
| **Code Review 依赖人工触发** | 高 | 集成自动 Code Review，subagent 返回后强制审查 |

#### 2. **技能系统问题**

| 问题 | 严重性 | 改进建议 |
|------|--------|----------|
| **技能发现机制低效** | 中 | 实现技能注册表（registry.json），避免每次遍历文件系统 |
| **技能依赖无版本控制** | 高 | 添加技能版本依赖解析（如 `requires: [two-agent-loop@^1.0.0]`） |

| **技能触发器过于简单** | 低 | 使用 LLM 进行语义匹配，而非关键词匹配 |

#### 3. **存储与数据管理**

| 问题 | 严重性 | 改进建议 |

| **JSON 文件易损坏** | 高 | 引入 SQLite 或 LMDB 替代 JSON 文件存储 |

| **跨 CLI 数据同步竞争** | 高 | 引入文件锁或分布式锁机制 |
