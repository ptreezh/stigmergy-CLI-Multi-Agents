# Stigmergy 架构分析报告

## 一、架构概述

**Stigmergy** 是一个多 AI CLI 协作系统，基于"痕迹刺激"(Stigmergy) 概念，实现多个 AI 助手之间的间接协作。当前版本：**1.2.1**

### 核心设计理念

1. **痕迹协作**: AI 通过共享文件/技能系统留下"痕迹"，其他 AI 读取并响应
2. **技能驱动**: 通过 `.md` 格式的技能文件定义 AI 行为模式
3. **跨 CLI 通信**: 支持 8+ AI CLI 工具之间的协作 (Claude/Gemini/Qwen/iFlow 等)

## 三、架构问题识别

### 🔴 严重问题

1. **技能发现路径冗余**
   ```json
   "skill_discovery_paths": [
     "C:\\Users\\WIN10\\.stigmergy\\skills",
     "C:\\Users\\WIN10\\.qwen\\skills",
     "C:\\stigmergy-CLI-Multi-Agents\\skills",
     "C:\\Users\\WIN10\\.stigmergy\\skills\\skills",  // 重复
     "C:\\Users\\WIN10\\.qwen\\skills\\skills"        // 重复
   ]
   ```

2. **安全规则矛盾**
   - `blocked_patterns` 包含 `"superpowers:.*"` 但技能系统依赖 superpowers 调用
   - `qwen_safe_executor.js` 有硬编码的 `safeSuperpowers` 列表，但未与技能系统同步

3. **Hook 系统碎片化**
   - `session-start.ts` 使用硬编码路径 `path.join(os.homedir(), 'skills')`
   - 未使用配置的 `SKILLS_ROOT` 路径

4. **技能重复存储**
   - `~/.qwen/skills/` 和 `~/.qwen/skills/skills/` 内容重复
   - `extensions/superpowers-qwen/skills/` 又有另一份副本

### 🟡 中等问题

5. **缺少集中式技能注册表**
   - 技能发现依赖文件系统扫描
   - 无技能版本控制/依赖管理

6. **跨 CLI 通信机制原始**
   - 依赖 `stigmergy distcli` shell 命令
   - 无消息队列/事件总线

7. **配置分散**
   - `config.json`, `hooks.json`, 各技能内配置
   - 无配置验证/迁移机制

8. **错误处理不统一**
   - 各 Hook 独立错误处理
   - 无全局错误追踪

### 🟢 轻微问题

9. **文档分散**
   - 200+ 个 `.md` 文件在根目录
   - 缺少文档索引/导航

10. **技能加载效率**
    - 每次会话加载所有技能元数据
    - 无懒加载/缓存机制

#### 2. 重构安全层

```javascript
// 建议：动态安全规则生成
class DynamicSafetyManager {
  constructor() {
    this.skillRegistry = this.loadSkillRegistry();
    this.safetyRules = this.generateSafetyRules();
  }
  
  generateSafetyRules() {
    // 从技能注册表动态生成允许列表
    const allowedSkills = this.skillRegistry
      .filter(s => s.verified)
      .map(s => s.name);
    
    return {
      allowedSuperpowers: allowedSkills,
      blockedPatterns: this.loadBlockedPatterns()
    };
  }
}
```
