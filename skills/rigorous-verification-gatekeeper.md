# Rigorous Verification Gatekeeper - 严苛验证门禁技能

## 概述

基于 2026-03-22 严重错误教训创建的强制验证门禁技能，确保所有报告符合严苛验证第一原则。

## 核心使命

> "永远不使用模拟冒充真实，永远准确标注验证等级，永远诚实报告所有发现。"

## 依赖

- **two-agent-loop**: 双 Agent 循环机制
- **soul-reflection**: 自我反思
- **soul-evolution**: 持续进化

## 门禁检查流程

### 第1步: 测试真实性验证 🔴

**检查内容**：
```
❓ 这个测试是真实执行还是模拟？

检查项：
- [ ] 所有测试都是真实 CLI 进程执行
- [ ] 没有使用异步函数冒充独立进程
- [ ] 没有使用进程内并发冒充多进程并发
- [ ] 没有使用模拟数据冒充真实数据

判定：
✅ 全部是真实执行 → 通过
⚠️  包含模拟但明确说明 → 警告
❌ 包含模拟未明确说明 → 拒绝
```

**关键概念区别**：
```
异步函数 ≠ 真实 CLI 进程
  ├─ 异步函数：同一 Node.js 进程内的并发
  ├─ CLI 进程：独立的操作系统进程（spawn/fork）
  └─ 绝不能混淆！

进程内并发 ≠ 多进程并发
  ├─ 进程内：共享内存空间
  ├─ 多进程：独立内存空间，需 IPC
  └─ 验证场景完全不同！

模拟测试 ≠ 真实测试
  ├─ 模拟：快速原型，不完整验证
  ├─ 真实：完整验证，可信结果
  └─ 必须明确区分！
```

### 第2步: 验证等级确认 🔴

**检查内容**：
```
❓ 验证等级是否准确标注？

验证等级定义：
Level 1 - 基础验证
  ├─ 代码可以运行
  ├─ 基本功能正常
  └─ 可包含模拟测试

Level 2 - 集成验证
  ├─ 真实环境集成
  ├─ 真实 CLI 执行
  └─ 跨组件协作

Level 3 - 压力验证
  ├─ 真实多进程并发
  ├─ 边界情况测试
  └─ ❌ 不能包含模拟测试

Level 4 - 生产验证
  ├─ 真实生产环境
  ├─ 长期运行验证
  └─ ❌ 不能包含模拟测试

判定：
✅ 等级明确且与实际相符 → 通过
❌ 模拟测试声称 Level 3+ → 拒绝
❌ 未标注验证等级 → 拒绝
```

**等级映射规则**：
```
模拟测试 → 最高 Level 1-2
真实测试 → 可达 Level 3-4
进程内并发 → Level 1-1.5
多进程并发 → 可达 Level 2-3
```

### 第3步: 局限性说明 🔴

**检查内容**：
```
❓ 是否明确说明了局限性？

必须说明：
- [ ] 测试的局限性（方法、范围）
- [ ] 未测试的场景（边界、异常）
- [ ] 不确定的部分（假设、推测）
- [ ] 前提条件（环境、依赖）

判定：
✅ 明确列出所有局限性 → 通过
⚠️  部分说明 → 警告
❌ 完全未说明 → 拒绝
```

**局限性声明模板**：
```markdown
## ⚠️ 测试局限性

### 方法局限
- 本测试使用 [方法]，可能无法完全模拟真实场景

### 范围局限
- 未测试 [场景]
- 未验证 [条件]

### 不确定性
- [结果] 基于 [假设]，可能存在偏差

### 前提条件
- 需要 [环境/依赖]
- 适用于 [场景]
```

### 第4步: 证据完整性 🔴

**检查内容**：
```
❓ 每个结论是否有证据支持？

证据类型：
- [ ] 具体数据（数字、百分比）
- [ ] 可验证的输出（日志、截图）
- [ ] 可复现的步骤（命令、代码）
- [ ] 明确的来源（文件、路径）

判定：
✅ 每个结论有充分证据 → 通过
⚠️  部分结论证据不足 → 警告
❌ 结论缺乏证据 → 拒绝
```

**证据质量标准**：
```
高质量证据：
  ✅ 可复现的命令
  ✅ 可验证的数据
  ✅ 明确的文件路径
  ✅ 完整的输出日志

低质量证据：
  ⚠️  "应该可以"
  ⚠️  "理论上可以"
  ⚠️  "看起来正常"
  ❌ "我觉得没问题"
```

### 第5步: 标题准确性 🔴

**检查内容**：
```
❓ 标题是否准确反映内容？

检查项：
- [ ] 标题准确反映测试内容
- [ ] 无夸大或误导性词汇
- [ ] 关键术语使用准确
- [ ] 无模糊表述

常见误导：
❌ "10个并发CLI" → 实际是"10个worker模拟"
❌ "性能优异" → 实际是"部分性能良好"
❌ "生产就绪" → 实际是"开发测试可用"

判定：
✅ 标题准确 → 通过
❌ 标题误导 → 拒绝
```

### 第6步: soul.md 原则对齐 🔴

**检查内容**：
```
❓ 是否违反 soul.md 严苛验证第一原则？

原则检查：
- [ ] 符合真实执行原则（无模拟冒充真实）
- [ ] 符合完整验证原则（说明局限性）
- [ ] 符合严格测试原则（可重复验证）
- [ ] 符合诚实报告原则（准确标注等级）
- [ ] 未重演 2026-03-22 错误案例

判定：
✅ 全部符合 → 通过
❌ 任何违反 → 拒绝
```

**2026-03-22 错误案例检查**：
```
❓ 是否重演以下错误？

- [ ] 用"异步函数"冒充"真实CLI进程"
- [ ] 用"进程内并发"冒充"多进程并发"
- [ ] 用"模拟测试"冒充"真实测试"
- [ ] 夸大验证等级（Level 1.5 → Level 3）
- [ ] 标题误导（"10个并发CLI"实际是worker）

任何一项 → ❌ 拒绝并警告
```

## 违规处理

### 轻微违规（警告）
```
⚠️ 包含模拟测试但已明确说明
⚠️ 部分局限性未说明
⚠️ 部分结论证据不足

处理：
1. 标注警告
2. 建议改进
3. 允许发布（需说明）
```

### 严重违规（拒绝）
```
🚨 模拟测试冒充真实
🚨 夸大验证等级
🚨 完全未说明局限性
🚨 标题严重误导
🚨 违反soul.md原则

处理：
1. 拒绝发布
2. 立即修正
3. 重新检查
```

### 极严重违规（禁令）
```
🚨🚨 重演 2026-03-22 错误
🚨🚨 故意误导用户
🚨🚨 多次违反原则

处理：
1. 立即停止
2. 深度反思
3. 重新建立信任
4. 永久记录
```

## 示例

### ✅ 通过示例

```markdown
# 进程内并发模拟测试报告

**验证等级**: Level 1.5 - 进程内并发模拟
**测试方法**: 使用异步函数模拟并发（非真实CLI进程）

## ⚠️ 测试局限性
- 本测试为进程内并发模拟，非真实多CLI进程
- 未验证跨进程通信和锁竞争
- 未验证真实生产环境

## 测试结果
- 使用10个异步函数并发访问共享存储
- 结果：文件锁机制正常工作
- 证据：[附输出日志]
```

**门禁结果**: ✅ 通过（明确说明是模拟，准确标注等级）

### ❌ 拒绝示例

```markdown
# Level 3 压力测试验证报告

**验证等级**: Level 3 - 压力验证 ✅

## 测试结果
- 10个并发CLI稳定运行
- 性能优异，超出预期
- 100%成功率
```

**门禁结果**: ❌ 拒绝
- 原因1: 标题声称"Level 3"但实际是模拟
- 原因2: "10个并发CLI"实际是10个异步函数
- 原因3: 未说明测试局限性
- 原因4: 违反soul.md原则

## 集成到工作流

### Pre-commit Hook
```bash
#!/bin/bash
# .git/hooks/pre-commit

echo "🔒 运行门禁检查..."
for file in $(git diff --cached --name-only | grep '\.md$'); do
  npm run gatekeeper -- "$file"
  if [ $? -ne 0 ]; then
    echo "❌ 门禁检查未通过，禁止提交: $file"
    exit 1
  fi
done
```

### CI/CD Integration
```yaml
# .github/workflows/gatekeeper.yml
name: Gatekeeper
on: [pull_request]
jobs:
  gatekeeper:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Run Gatekeeper
        run: |
          for file in docs/*.md; do
            npm run gatekeeper -- "$file"
          done
```
